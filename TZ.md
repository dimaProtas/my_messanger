# Техническое задание: Backend упрощённого мессенджера (аналог Telegram)

---

## 1. Общее описание проекта

### 1.1. Что строим

Backend-часть упрощённого мессенджера, подобного Telegram. Система обеспечивает:

- Регистрацию и аутентификацию пользователей по номеру телефона и паролю с использованием JWT (Access + Refresh токены).
- Обмен текстовыми сообщениями в личных и групповых чатах.
- Доставку сообщений в реальном времени через WebSocket с использованием Redis Pub/Sub.
- Асинхронную обработку сообщений через Kafka для надёжности и масштабирования.
- Управление контактами и профилем пользователя.

### 1.2. Цели

| Цель | Описание |
|------|----------|
| **MVP-мессенджер** | Работающий backend с минимальным, но полным набором функций для общения |
| **Надёжная доставка** | Гарантированная доставка сообщений через Kafka (at-least-once семантика) |
| **Реальное время** | Мгновенная доставка онлайн-пользователям через WebSocket + Redis Pub/Sub |
| **Чистая архитектура** | Строгое разделение на слои (handler → service → repository) для поддерживаемости |
| **Готовность к расширению** | Возможность добавления файлов, голосовых сообщений, шифрования в будущем |

### 1.3. Стек технологий (уже выбран и зафиксирован)

| Компонент | Технология | Назначение |
|-----------|------------|------------|
| Язык | Go 1.26.2 | Бэкенд |
| HTTP-роутер | chi/v5 | REST API |
| База данных | PostgreSQL + lib/pq | Основное хранилище |
| Кэш / Pub/Sub | Redis (go-redis/v8) | Сессии, уведомления в реальном времени |
| Очередь сообщений | Kafka (segmentio/kafka-go) | Асинхронная обработка сообщений |
| WebSocket | gorilla/websocket | Доставка в реальном времени |
| Аутентификация | golang-jwt/jwt/v5 (HS256) | Access/Refresh токены |
| Конфигурация | godotenv | Переменные окружения из `.env` |

## 2. Функциональные требования

### 2.1. Аутентификация и регистрация

#### 2.1.1. Регистрация пользователя

| Параметр | Значение |
|----------|----------|
| **Приоритет** | P0 — критический |
| **Метод** | `POST /api/v1/auth/register` |
| **Входные данные** | `phone_number` (строка, формат E.164 или цифры), `password` (строка, мин. 8 символов), `username` (строка, 3–50 символов) |
| **Выходные данные** | `access_token`, `refresh_token`, `user` (id, username, phone_number, created_at) |
| **Критерии приёмки** | 1. Пароль хэшируется bcrypt (cost ≥ 12) перед сохранением. 2. Номер телефона нормализуется (удаляются пробелы, скобки, дефисы). 3. Нельзя зарегистрироваться с уже существующим номером — ошибка 409 Conflict. 4. Refresh-токен сохраняется в БД и Redis. 5. Access-токен возвращается в теле ответа. |

#### 2.1.2. Вход в систему

| Параметр | Значение |
|----------|----------|
| **Приоритет** | P0 — критический |
| **Метод** | `POST /api/v1/auth/login` |
| **Входные данные** | `phone_number`, `password` |
| **Выходные данные** | `access_token`, `refresh_token`, `user` |
| **Критерии приёмки** | 1. Неверный номер или пароль — ошибка 401 Unauthorized. 2. При успешном входе создаётся новая пара токенов. 3. Старые refresh-токены пользователя (если есть) инвалидируются (опционально: или сохраняются для multiple sessions). |

#### 2.1.3. Обновление токенов

| Параметр | Значение |
|----------|----------|
| **Приоритет** | P0 — критический |
| **Метод** | `POST /api/v1/auth/refresh` |
| **Входные данные** | `refresh_token` |
| **Выходные данные** | `access_token`, `refresh_token` (новая пара) |
| **Критерии приёмки** | 1. Refresh-токен должен существовать в БД и Redis, быть не истёкшим и не отозванным. 2. Старый refresh-токен инвалидируется (удаляется). 3. Выдаётся новая пара токенов. 4. Если refresh-токен не найден или истёк — ошибка 401. |

#### 2.1.4. Выход из системы

| Параметр | Значение |
|----------|----------|
| **Приоритет** | P1 — важный |
| **Метод** | `POST /api/v1/auth/logout` |
| **Входные данные** | `refresh_token` (в теле запроса) + Access-токен в заголовке `Authorization` |
| **Выходные данные** | `204 No Content` |
| **Критерии приёмки** | 1. Refresh-токен удаляется из БД и Redis. 2. Access-токен добавляется в blacklist в Redis с TTL = оставшееся время жизни. 3. Все сессии пользователя могут быть завершены опционально. |

### 2.2. Профиль пользователя

#### 2.2.1. Получение профиля

| Параметр | Значение |
|----------|----------|
| **Приоритет** | P1 |
| **Метод** | `GET /api/v1/users/me` |
| **Аутентификация** | Access-токен в заголовке `Authorization: Bearer <token>` |
| **Выходные данные** | `id`, `phone_number`, `username`, `created_at`, `updated_at` |
| **Критерии приёмки** | 1. Данные возвращаются только для аутентифицированного пользователя. 2. `phone_number` маскируется (видны только последние 4 цифры) всем, кроме самого владельца. |

#### 2.2.2. Обновление профиля

| Параметр | Значение |
|----------|----------|
| **Приоритет** | P2 |
| **Метод** | `PUT /api/v1/users/me` |
| **Входные данные** | `username` (опционально) |
| **Выходные данные** | Обновлённый объект `user` |
| **Критерии приёмки** | 1. `username` должен быть уникальным в системе (если меняется). 2. Поле `updated_at` обновляется автоматически. |

#### 2.2.3. Поиск пользователей по номеру телефона

| Параметр | Значение |
|----------|----------|
| **Приоритет** | P1 |
| **Метод** | `GET /api/v1/users/search?phone=...` |
| **Выходные данные** | Массив `[{id, username, phone_number_masked}]` |
| **Критерии приёмки** | 1. Поиск по частичному совпадению номера. 2. Возвращается не более 20 результатов. 3. Сам пользователь исключается из результатов. |

### 2.3. Управление контактами

#### 2.3.1. Добавление контакта

| Параметр | Значение |
|----------|----------|
| **Приоритет** | P1 |
| **Метод** | `POST /api/v1/contacts` |
| **Входные данные** | `contact_user_id` (UUID) |
| **Выходные данные** | Объект контакта `{id, user_id, contact_user_id, created_at}` |
| **Критерии приёмки** | 1. Нельзя добавить самого себя — ошибка 400. 2. Нельзя добавить уже существующий контакт — ошибка 409. 3. `contact_user_id` должен существовать в таблице `users` — иначе 404. 4. Контакт добавляется однонаправленно (не требует подтверждения). |

#### 2.3.2. Удаление контакта

| Параметр | Значение |
|----------|----------|
| **Приоритет** | P1 |
| **Метод** | `DELETE /api/v1/contacts/{contactId}` |
| **Выходные данные** | `204 No Content` |
| **Критерии приёмки** | 1. Удалить можно только свой контакт. 2. Несуществующий контакт — 404. |

#### 2.3.3. Список контактов

| Параметр | Значение |
|----------|----------|
| **Приоритет** | P1 |
| **Метод** | `GET /api/v1/contacts` |
| **Выходные данные** | Массив объектов `[{id, contact_user: {id, username, phone_number_masked}, created_at}]` |
| **Критерии приёмки** | 1. Возвращаются только контакты текущего пользователя. 2. Данные контакта включают username и маскированный телефон. |

### 2.4. Чаты

#### 2.4.1. Создание личного чата (private)

| Параметр | Значение |
|----------|----------|
| **Приоритет** | P0 |
| **Метод** | `POST /api/v1/chats/private` |
| **Входные данные** | `user_id` (UUID собеседника) |
| **Выходные данные** | Объект чата `{id, type: "private", members: [...], created_at}` |
| **Критерии приёмки** | 1. Если личный чат между двумя пользователями уже существует — вернуть его (не создавать дубликат). 2. Оба пользователя автоматически становятся участниками. 3. Нельзя создать чат с самим собой — 400. |

#### 2.4.2. Создание группового чата (group)

| Параметр | Значение |
|----------|----------|
| **Приоритет** | P0 |
| **Метод** | `POST /api/v1/chats/group` |
| **Входные данные** | `name` (строка), `member_ids` (массив UUID, минимум 2 участника помимо создателя) |
| **Выходные данные** | Объект чата `{id, type: "group", name, members: [...], created_at}` |
| **Критерии приёмки** | 1. Создатель автоматически добавляется как участник. 2. Требуется добавить поле `name` в таблицу `chats` (миграция 000002). 3. Все `member_ids` должны существовать — иначе 404. 4. Минимум 2 других участника — иначе 400. |

#### 2.4.3. Список чатов пользователя

| Параметр | Значение |
|----------|----------|
| **Приоритет** | P0 |
| **Метод** | `GET /api/v1/chats` |
| **Выходные данные** | Массив объектов чата с последним сообщением и временем |
| **Критерии приёмки** | 1. Чаты сортируются по времени последнего сообщения (сначала новые). 2. Для каждого чата возвращается превью последнего сообщения. 3. Для личных чатов в качестве названия используется username собеседника. |

#### 2.4.4. Добавление участника в групповой чат

| Параметр | Значение |
|----------|----------|
| **Приоритет** | P1 |
| **Метод** | `POST /api/v1/chats/{chatId}/members` |
| **Входные данные** | `user_id` (UUID) |
| **Выходные данные** | `204 No Content` |
| **Критерии приёмки** | 1. Только для group-чатов. 2. Добавлять может любой участник чата. 3. Нельзя добавить уже существующего участника — 409. 4. Пользователь должен существовать — 404. |

#### 2.4.5. Удаление участника из группового чата

| Параметр | Значение |
|----------|----------|
| **Приоритет** | P2 |
| **Метод** | `DELETE /api/v1/chats/{chatId}/members/{userId}` |
| **Выходные данные** | `204 No Content` |
| **Критерии приёмки** | 1. Только для group-чатов. 2. Удалять может только создатель чата (требуется поле `created_by` в `chats`). 3. Нельзя удалить самого создателя (опционально). |

### 2.5. Сообщения

#### 2.5.1. Отправка сообщения

| Параметр | Значение |
|----------|----------|
| **Приоритет** | P0 |
| **Метод** | `POST /api/v1/chats/{chatId}/messages` |
| **Входные данные** | `content` (текст, 1–4096 символов) |
| **Выходные данные** | Объект сообщения `{id, chat_id, sender_id, content, sent_at, status: "sent"}` |
| **Критерии приёмки** | 1. Отправитель должен быть участником чата — иначе 403. 2. Сообщение сохраняется в Kafka (топик `messages`). 3. Consumer читает из Kafka и сохраняет в PostgreSQL. 4. После сохранения в БД публикуется событие в Redis Pub/Sub в каналы `messages:{userID}` для всех участников чата. 5. Статус сообщения — `sent`. |

#### 2.5.2. Получение истории сообщений чата

| Параметр | Значение |
|----------|----------|
| **Приоритет** | P0 |
| **Метод** | `GET /api/v1/chats/{chatId}/messages` |
| **Параметры запроса** | `limit` (по умолчанию 50, максимум 100), `before` (UUID сообщения, для курсорной пагинации) |
| **Выходные данные** | Массив сообщений, отсортированных по `sent_at DESC` |
| **Критерии приёмки** | 1. Пагинация курсорная (по `sent_at` + `id` для разрешения коллизий времени). 2. Пользователь должен быть участником чата — иначе 403. 3. Возвращается не более `limit` сообщений. |

#### 2.5.3. Обновление статуса сообщения

| Параметр | Значение |
|----------|----------|
| **Приоритет** | P1 |
| **Метод** | `PUT /api/v1/messages/{messageId}/status` |
| **Входные данные** | `status` — `"delivered"` или `"read"` |
| **Выходные данные** | Обновлённый объект сообщения |
| **Критерии приёмки** | 1. Изменить статус может только получатель (не отправитель). 2. Переход статусов: `sent → delivered → read`. Нельзя перепрыгнуть. 3. `delivered` выставляется автоматически при доставке через WebSocket (или вручную клиентом). 4. Статус `read` выставляется клиентом при открытии чата. |

### 2.6. WebSocket-уведомления

| Параметр | Значение |
|----------|----------|
| **Приоритет** | P0 |
| **Эндпоинт** | `GET /api/v1/ws` (Upgrade до WebSocket) |
| **Аутентификация** | Access-токен через query-параметр `?token=...` |
| **Критерии приёмки** | 1. При подключении клиент подписывается на Redis-канал `messages:{userID}`. 2. Новые сообщения доставляются мгновенно в формате JSON. 3. WebSocket поддерживает Ping/Pong для поддержания соединения. 4. При разрыве соединения ресурсы освобождаются (канал Pub/Sub закрывается). 5. Один пользователь может иметь несколько WebSocket-соединений (разные устройства). |

## 3. Нефункциональные требования

### 3.1. Производительность

| Требование | Значение |
|------------|----------|
| Время ответа REST API (P95) | ≤ 200 мс (без учёта Kafka-задержки) |
| Время доставки сообщения через WebSocket (P95) | ≤ 500 мс от отправки до получения |
| Пропускная способность | ≥ 1000 сообщений/сек на одном инстансе |
| Количество одновременных WebSocket-соединений | ≥ 10 000 на инстанс |
| Размер пула соединений PostgreSQL | 25 активных, 5 idle |

### 3.2. Безопасность

| Требование | Реализация |
|------------|------------|
| Хэширование паролей | bcrypt, cost = 12 |
| Access-токен | JWT HS256, TTL = 1 час |
| Refresh-токен | JWT HS256, TTL = 7 дней, хранится в БД + Redis |
| Blacklist токенов | Redis, ключ `blacklist:{jti}` с TTL до истечения токена |
| CORS | Настраиваемый список разрешённых origin |
| Rate limiting | На уровне middleware (chi), 100 запросов/мин на пользователя |
| Валидация входных данных | Все поля валидируются: длина, формат, диапазон |
| SQL-инъекции | Исключены использованием параметризованных запросов (`$1`, `$2`) |

### 3.3. Отказоустойчивость

| Требование | Реализация |
|------------|------------|
| Graceful shutdown | HTTP-сервер завершает приём новых запросов, ожидает завершения текущих (таймаут 30 сек) |
| Переподключение к БД | Автоматическое (connection pool + retry) |
| Переподключение к Redis | Автоматическое (клиент go-redis) |
| Переподключение к Kafka | Автоматическое (клиент segmentio/kafka-go) |
| Health-check | `GET /health` — проверка подключения к БД, Redis, Kafka |
| Readiness-check | `GET /ready` — готовность принимать трафик |

### 3.4. Логирование и мониторинг

| Требование | Реализация |
|------------|------------|
| Структурированное логирование | `log/slog` (стандартная библиотека Go) |
| Уровни логирования | DEBUG, INFO, WARN, ERROR |
| Trace ID | X-Request-ID передаётся через контекст во все слои |
| Метрики | Prometheus-метрики для HTTP (опционально, этап 3) |

### 3.5. Масштабирование

| Требование | Реализация |
|------------|------------|
| Stateless HTTP | Вся сессионная информация в токенах и Redis |
| Горизонтальное масштабирование | Несколько экземпляров за балансировщиком |
| Kafka consumer group | Все инстансы в одной consumer group для балансировки |
| Redis Pub/Sub | Каждый инстанс подписывается на каналы подключённых пользователей |

          ┌─────────────────┼─────────────────┐
          ▼                 ▼                 ▼
┌─────────────────┐ ┌─────────────┐ ┌─────────────────┐

│                  │ │ JWT,Kafka   │ │                  │
└─────────────────┘ └─────────────┘ └─────────────────┘
          │
          ▼
┌─────────────────────────────────────────────────────────┐
│                internal/{domain}                         │
│  ┌──────────┐    ┌──────────┐    ┌──────────────┐      │

```json
{
  "type": "message_status",
  "payload": {
    "message_id": "aa0e8400-...",
    "chat_id": "880e8400-...",
    "status": "delivered",
    "updated_at": "2026-06-14T12:03:30Z"
  }
}
```

От клиента серверу (отправка сообщения через WS — опционально, этап 2):
```json
{
  "type": "send_message",
  "payload": {
    "chat_id": "880e8400-...",
    "content": "Привет!"
  }
}
```

От клиента серверу (пользователь печатает — опционально):
```json
{
  "type": "typing",
  "payload": {
    "chat_id": "880e8400-..."
  }
}
```

**Управление соединением:**
- Ping каждые 54 секунды (сервер → клиент)
- Pong ожидается в течение 60 секунд, иначе разрыв
- Максимальный размер входящего сообщения: 4096 байт (текстовые сообщения)
- При разрыве соединения клиент должен переподключиться с новым токеном

## 6. Схема данных

### 6.1. Существующая схема (миграция 000001)

Существующая миграция `migrations/000001_initial_schema.up.sql` содержит таблицы `users`, `contacts`, `chats`, `chat_members`, `messages` (см. контекст). Детально — в файле миграции.

### 6.2. Необходимые расширения схемы (миграция 000002)

```sql
-- migrations/000002_schema_additions.up.sql

-- 1. Таблица для хранения refresh-токенов
CREATE TABLE refresh_tokens (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    token_hash BYTEA NOT NULL,         -- bcrypt-хэш refresh-токена для поиска
    jti VARCHAR(36) NOT NULL UNIQUE,   -- JWT ID токена (UUID)
    expires_at TIMESTAMPTZ NOT NULL,
    created_at TIMESTAMPTZ DEFAULT NOW(),
    revoked_at TIMESTAMPTZ              -- NULL если активен, иначе время отзыва
);

CREATE INDEX idx_refresh_tokens_user_id ON refresh_tokens(user_id);
CREATE INDEX idx_refresh_tokens_jti ON refresh_tokens(jti);
CREATE INDEX idx_refresh_tokens_expires_at ON refresh_tokens(expires_at);

-- 2. Добавление поля name в таблицу chats (для групповых чатов)
ALTER TABLE chats ADD COLUMN name VARCHAR(100);

-- 3. Добавление поля created_by в таблицу chats (создатель группового чата)
ALTER TABLE chats ADD COLUMN created_by UUID REFERENCES users(id);

-- 4. Добавление поля updated_at в таблицу messages (для статусов)
ALTER TABLE messages ADD COLUMN updated_at TIMESTAMPTZ DEFAULT NOW();

-- 5. Индекс для поиска личного чата между двумя пользователями
CREATE INDEX idx_chat_members_chat_user ON chat_members(chat_id, user_id);
```

Соответствующая down-миграция:
```sql
-- migrations/000002_schema_additions.down.sql

DROP INDEX IF EXISTS idx_chat_members_chat_user;
ALTER TABLE messages DROP COLUMN IF EXISTS updated_at;
ALTER TABLE chats DROP COLUMN IF EXISTS created_by;
ALTER TABLE chats DROP COLUMN IF EXISTS name;
DROP TABLE IF EXISTS refresh_tokens;
```

## 7. Поток сообщений (детально)

### 7.1. Отправка сообщения (полный путь)

```
Шаг 1: Клиент отправляет POST /api/v1/chats/{chatId}/messages
       с заголовком Authorization: Bearer <access_token>

Шаг 2: Middleware AuthMiddleware извлекает и валидирует access-токен.
       Из JWT-claims извлекается userID.
       Проверяется blacklist в Redis (ключ blacklist:{jti}).
       Если токен невалиден/отозван → 401.
       userID и requestID помещаются в context.

Шаг 3: message.Handler.SendMessage(ctx, chatID, content)
       │
       ├─ 3a. chat.Service.IsMember(ctx, chatID, userID)
       │      └─ chat.Repository → SQL: SELECT 1 FROM chat_members
       │         WHERE chat_id=$1 AND user_id=$2
       │      Если не участник → 403.
       │
       ├─ 3b. message.Service.Send(ctx, msg)
       │      │
       │      ├─ 3b1. Генерация UUID для сообщения (msg.ID)
       │      ├─ 3b2. Сериализация Message в JSON
       │      └─ 3b3. Kafka Producer → топик "messages"
       │             ключ: chatID, значение: JSON
       │
       └─ 3c. Возврат 202 Accepted с объектом сообщения
              (статус "sent", клиент получает мгновенный ответ)

Шаг 4: Kafka (асинхронно)
       Сообщение попадает в партицию топика "messages".
       Гарантии: at-least-once (acks=1, retries=3).

Шаг 5: message.Consumer (в отдельной горутине, запускается в main.go)
       │
       ├─ 5a. Читает батч сообщений из Kafka
       ├─ 5b. Десериализует JSON → Message
       ├─ 5c. Вызывает message.Service.ProcessMessage(ctx, msg)
       │      │
       │      ├─ 5c1. message.Repository.Save(ctx, msg)
       │      │       └─ SQL: INSERT INTO messages (id, chat_id, sender_id,
       │      │              content, sent_at, status)
       │      │              VALUES ($1, $2, $3, $4, $5, 'sent')
       │      │
       │      ├─ 5c2. chat.Repository.GetMemberIDs(ctx, msg.ChatID)
       │      │       └─ SQL: SELECT user_id FROM chat_members WHERE chat_id=$1
       │      │
       │      └─ 5c3. Для каждого memberID (кроме sender_id):
       │              redis.Publish(ctx, "messages:{memberID}", jsonMsg)
       │
       └─ 5d. Коммит offset в Kafka

Шаг 6: Redis Pub/Sub
       Сообщение публикуется в каналы всех участников чата.

Шаг 7: WebSocket (server/websocket.go → writePump)
       │
       ├─ 7a. Горутина writePump подписана на канал "messages:{userID}"
       ├─ 7b. При получении сообщения из Pub/Sub канала:
       │      записывает JSON в WebSocket-соединение
       └─ 7c. Клиент получает сообщение в реальном времени

Шаг 8 (опционально): Клиент отправляет обновление статуса
       PUT /api/v1/messages/{messageId}/status {"status": "delivered"}
       → Kafka "message-status" → Consumer → PostgreSQL UPDATE → Redis Pub/Sub
```

### 7.2. Доставка офлайн-пользователям

1. Сообщение всегда сохраняется в PostgreSQL (шаг 5c1).
2. При следующем открытии чата клиент запрашивает историю через `GET /api/v1/chats/{chatId}/messages`.
3. Непрочитанные сообщения подсчитываются по статусу (`sent` или `delivered`, но не `read`) для всех сообщений, адресованных пользователю.

## 10. Исправление текущих багов (критические)

### 10.1. config/config.go — функция `splitAndTrim`

**Проблема:** Функция `splitAndTrim` использует `os.SplitList` вместо `strings.Split`, что заставляет её игнорировать переданный разделитель `sep` и использовать `os.PathListSeparator`. Также `os.ExpandEnv` вместо `strings.TrimSpace` приводит к некорректной обработке значений.

**Исправление:**
```go
// Было:
func splitAndTrim(s string, sep string) []string {
    var result []string
    parts := os.SplitList(s)
    for _, p := range parts {
        if t := os.ExpandEnv(p); t != "" {
            result = append(result, t)
        }
    }
    return result
}

// Должно быть:
func splitAndTrim(s string, sep string) []string {
    var result []string
    parts := strings.Split(s, sep)
    for _, p := range parts {
        if t := strings.TrimSpace(p); t != "" {
            result = append(result, t)
        }
    }
    return result
}
```

Убрать импорт `"os"`, добавить импорт `"strings"` (или `os` всё ещё нужен для `os.LookupEnv` и `os.Getenv` — проверить).

### 10.2. server/websocket.go — отсутствует импорт fmt

**Проблема:** В строке 103 используется `fmt.Sprintf("messages:%s", userID)`, но пакет `"fmt"` не импортирован. Ошибка компиляции.

**Исправление:** Добавить `"fmt"` в секцию импорта.

### 10.3. server/websocket.go — хардкод userID

**Проблема:** Строка 60: `userID := "some-user-id"`. Не извлекается из JWT.

**Исправление:** Реализовать извлечение userID из JWT (будет сделано на этапе 6). До этого момента оставить заглушку с комментарием, но раскомментировать после реализации middleware.

### 10.4. server/websocket.go — CheckOrigin всегда true

**Проблема:** Строка 34: `return true`. Потенциальная уязвимость CSRF через WebSocket.

**Исправление:** Добавить поле `AllowedOrigins []string` в конфиг и структуру `WsServer`. Проверять `r.Header.Get("Origin")` на вхождение в список разрешённых origin. Для разработки можно разрешить `http://localhost:*`.

### 10.5. pkg/jwt/jwt.go — JTI refresh-токена = userID

**Проблема:** В `GenerateRefreshToken` поле `ID` в `RegisteredClaims` устанавливается в `userID`. Это означает, что у пользователя может быть только ОДИН активный refresh-токен. Если он заходит с двух устройств, старый токен перезаписывается (одинаковый JTI).

**Исправление:**
```go
// Было:
ID: userID,

// Должно быть:
ID: uuid.New().String(),
```

Добавить импорт `"github.com/google/uuid"` (или использовать `crypto/rand` для генерации UUID). Также необходимо хранить JTI в Redis для быстрой проверки и в БД для персистентности.

### 10.6. pkg/kafka/producer.go — BatchTimeout без time.Millisecond

**Проблема:** Строка 21: `BatchTimeout: 10 * 1000000`. Значение корректное (10 миллионов наносекунд = 10 мс), но запись некорректная с точки зрения семантики. `BatchTimeout` ожидает `time.Duration`. Значение `10 * 1000000` — это беззнаковое целое (наносекунды), но лучше использовать константу.

**Исправление:**
```go
// Было:
BatchTimeout: 10 * 1000000,

// Должно быть:
BatchTimeout: 10 * time.Millisecond,
```

Добавить импорт `"time"` (если ещё не добавлен).

### 10.7. Дополнительно: опечатка в названии модуля

**Проблема:** В `go.mod` модуль назван `my_messanger` (опечатка — «messanger» вместо «messenger»).

**Решение:** Переименовать на `my_messenger` в `go.mod` и во всех импортах (все файлы, где используется `"my_messanger/..."`). Это технический долг, рекомендуется исправить на этапе 0, чтобы избежать путаницы.

## 11. Риски и компромиссы

### 11.1. Риски

| Риск | Вероятность | Влияние | Минимизация |
|------|-------------|---------|-------------|
| **Kafka недоступна** | Средняя | Сообщения не доставляются | Health-check + мониторинг; fallback: прямая запись в БД (на этапе MVP не реализуется) |
| **Redis потеря данных** | Низкая | Потеря Pub/Sub сообщений для онлайн-пользователей | Сообщения всегда сохраняются в PostgreSQL; при переподключении WebSocket клиент запрашивает историю |
| **Утечка WebSocket-соединений** | Средняя | Рост потребления памяти | Connection Manager с периодической очисткой «мёртвых» соединений; Ping/Pong |
| **Дубликаты сообщений из-за Kafka at-least-once** | Средняя | Дубликаты в БД и доставке | Идемпотентность по UUID сообщения: `INSERT ... ON CONFLICT (id) DO NOTHING` |
| **Рост таблицы refresh_tokens** | Высокая | Замедление запросов | Периодическая очистка (cron job): удаление истёкших и отозванных токенов старше N дней |
| **Конкурентное обновление статуса сообщения** | Низкая | Непредсказуемое состояние | `UPDATE ... WHERE status = $expectedStatus` (optimistic locking по статусу) |

### 11.2. Компромиссы

| Решение | Компромисс |
|---------|------------|
| **At-least-once вместо exactly-once** | Упрощение архитектуры (нет транзакций Kafka-to-DB). Дубликаты предотвращаются на уровне БД (первичный ключ UUID). |
| **Одна consumer group для всех инстансов** | Все инстансы читают из одной Kafka consumer group, партиции распределяются между ними. При масштабировании > числа партиций часть инстансов будет простаивать. |
| **Redis Pub/Sub без гарантий доставки** | Если подписчик офлайн — сообщение теряется в Pub/Sub. Это приемлемо, т.к. сообщение всегда можно получить через REST API (история). |
| **bcrypt для refresh-токенов** | Хэш bcrypt для поиска refresh-токена медленнее, чем SHA-256, но безопаснее. JTI (в открытом виде) используется для быстрого поиска. |
| **Проверка isMember при каждом запросе** | Запрос в БД при каждой отправке сообщения. Можно кэшировать в Redis, но на этапе MVP — прямой запрос (простота). |
| **Монолитная структура проекта** | Все домены в одном бинарнике. Для MVP это оправдано. При росте нагрузки можно разбить на микросервисы (auth, chat, message). |

## 13. Формат коммитов и Git-веток

Для разработки рекомендуется использовать следующий подход:

- **Ветки:** `feature/etap-N-opisanie` (например, `feature/etap-1-infrastructure`)
- **Коммиты:** conventional commits:
  - `feat(auth): add registration endpoint`
  - `fix(config): splitAndTrim uses strings.Split instead of os.SplitList`
  - `refactor(websocket): extract userID from JWT`
  - `test(auth): add service layer unit tests`

## 14. Приложение: стандартные коды ошибок HTTP

| Код | Код ошибки | Описание |
|-----|------------|----------|
| 400 | `INVALID_INPUT` | Ошибка валидации входных данных |
| 401 | `UNAUTHORIZED` | Токен отсутствует, невалиден или истёк |
| 401 | `TOKEN_REVOKED` | Токен отозван (logout) |
| 403 | `FORBIDDEN` | Нет прав на операцию (не участник чата) |
| 404 | `NOT_FOUND` | Ресурс не найден (чат, сообщение, пользователь) |
| 409 | `CONFLICT` | Конфликт (дубликат номера, username, контакта) |
| 429 | `RATE_LIMITED` | Превышен лимит запросов |
| 500 | `INTERNAL_ERROR` | Внутренняя ошибка сервера |
| 503 | `SERVICE_UNAVAILABLE` | Сервис временно недоступен |
